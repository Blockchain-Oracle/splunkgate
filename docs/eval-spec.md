## Datasets

### 1. JailbreakBench (`jailbreakbench` PyPI package)

100 harmful behaviors × 10 jailbreak variants = 1,000 prompts. Public, well-cited. Loaded via `from jailbreakbench import read_dataset`.

**Use:** binary classification benchmark for prompt-injection detection. Reports the prompt → harmful-behavior mapping.

**Coverage:** OWASP LLM01 Prompt Injection + LLM02 Sensitive Information Disclosure (`context/01-threat-landscape/05-owasp-llm-top-10.md`).

### 2. AdvBench (git submodule `inspiration/llm-attacks/`)

520 adversarial prompts. Standard baseline for jailbreak detection (Zou et al. 2023).

**Use:** scaling test for the Anthropic Many-Shot Jailbreaking power-law (per the now-grounded PDF in `sources/pdfs/anthropic-many-shot-jailbreaking-2024-04-02.pdf`). We measure detection rate as a function of in-context shot count to confirm we sit on the same scaling curve.

### 3. Imprompter payloads (custom, derived from arxiv 2410.14923v2)

11 reference payloads from the Imprompter paper (`sources/pdfs/imprompter-2410.14923v2.pdf`) targeting LeChat (Mistral) and ChatGLM. Verbatim payloads stored at `Synthetic-Data/pii_leak_corpus/imprompter_payloads.jsonl`.

**Use:** PII-exfil detection benchmark. Tests `check_output_leak` MCP tool + `model_middleware` post-emission scan.

**Note:** Mistral patched the rendering-layer exfil vector 2024-09-13 by disabling external markdown image rendering. Imprompter test corpus is for *detection*, not *prevention* — Aegis catches the payload pattern in the LLM output before the client-side renderer would have processed it.

### 4. Custom synthetic corpus (`Synthetic-Data/`)

Generated by `Synthetic-Data/generate_agent_verdicts.py`. Mirrors DNS Guard's pattern (`context/11-prior-art/01-build-a-thon-2025-deep-read.md`).

Three sub-corpora:
- **Tool-call abuse** (~200 prompts): adversarial tool arguments (path traversal, SQL injection in tool args, semantic shell injection)
- **Multi-turn injection** (~150 sequences): MSJ-style faux dialogue chains testing the power-law per the Anthropic paper
- **Benign control** (~300 prompts): real-world enterprise SOC analyst queries + customer support transcripts (anonymized + synthetic-only — zero real customer data per `context/04-pii-phi-pci/01-pii-definitions-by-jurisdiction.md`)

### 5. PII/PHI/PCI ground-truth corpus (`Synthetic-Data/pii_leak_corpus/`)

200 outputs labeled by jurisdiction:
- 50 GDPR personal data (Article 4(1) definition — verbatim from `context/04-pii-phi-pci/01-pii-definitions-by-jurisdiction.md`)
- 50 HIPAA PHI (Safe Harbor 18 identifiers from `context/04-pii-phi-pci/02-phi-hipaa-deep.md`)
- 50 PCI DSS Cardholder Data (CHD + SAD from `context/04-pii-phi-pci/03-pci-dss-cardholder-data.md`)
- 50 benign control

## Metrics

### Classification metrics (per evaluator × per dataset)

| Metric | Definition | Pass threshold | Notes |
|---|---|---|---|
| **Precision** | TP / (TP + FP) | ≥ 0.90 | Low false-block rate matters more than high detection for runtime use (don't block legitimate workflows). |
| **Recall** | TP / (TP + FN) | ≥ 0.85 | Reasonable detection rate. |
| **F1** | harmonic mean of P/R | ≥ 0.85 | Composite. |
| **Specificity** | TN / (TN + FP) | ≥ 0.95 | Direct measure of benign-input survival. |
| **ECE** (Expected Calibration Error) | per `context/lens-architect.md` discussion | ≤ 0.05 | Critical for runtime gates — confidence must match accuracy. Reliability diagram emitted as PNG to `eval/results/<sha>/reliability/`. |

### Latency metrics (per evaluator)

| Metric | Pass threshold | Notes |
|---|---|---|
| **p50 latency** | ≤ 500 ms | Real-time viable for synchronous gates. |
| **p99 latency** | ≤ 2 s | Worst-case acceptable for human-approval flows. |
| **Tail (p99.9)** | ≤ 5 s | Catches stragglers / retries. |

Per Splunk Agentic Ops customer expectations from `context/sources/docs-saved/skywork-mcp-deep-dive.md` and `context/09-personas-and-workflows/04-soc-analyst-day.md`.

### Cost metrics

| Metric | Pass threshold | Notes |
|---|---|---|
| **Cost per 1k verdicts** | report — no hard threshold | Comparative number across evaluators; published in README eval table. |

For Cisco AI Defense Inspection API: 10M queries/AI-app/year free (from `context/07-cisco-stack/01-ai-defense-deep.md`); we report effective $/1k as 0 within quota.

For gpt-oss-120b baseline: reported per Splunk Hosted Models pricing if we can confirm via Cloud trial; otherwise reported as "TBD pending dev-tier confirmation."

## Baselines (compare Aegis against these)

### Baseline 1 — DefenseClaw regex-only

Run the verdict pipeline with **only** DefenseClaw's regex rule packs (`internal/gateway/rules.go` from `cisco-ai-defense/defenseclaw` cloned at `inspiration/defenseclaw/`). No AI Defense API, no Foundation-Sec explanation. This is the floor — what you get from pattern matching alone.

Hypothesis: high precision, lower recall, very low cost, sub-50 ms latency. Misses semantic obfuscation (e.g., the Imprompter vocabulary-masking trick).

### Baseline 2 — gpt-oss-120b-as-judge

Replace Aegis's multi-model judgment chain with a single gpt-oss-120b call asking "is this prompt injection? classify the output for PII?" — generic LLM-as-judge pattern.

Hypothesis: balanced precision/recall, higher latency, higher cost, worse ECE than Cisco AI Defense (per the architect-lens reasoning — generic LLMs without domain training tend to be over-confident).

### Baseline 3 — Cisco AI Defense alone

Just the binary classifier. No Foundation-Sec explanation, no DefenseClaw second pass, no `splunklib/ai/security.py` first-pass.

Hypothesis: the precision/recall this is the ceiling for "use Cisco's product without us"; Aegis's value-add is the composition + the explanation layer + the SOC-integrated audit trail.

### Aegis full stack

`splunklib_security` first-pass → Cisco AI Defense classifier → Foundation-Sec explanation. Per `architecture.md` § "Judgment layer."

Hypothesis: precision dominated by AI Defense (composition is multiplicative on classifier quality); explanation quality dominated by Foundation-Sec; latency higher than baselines but worth it for the audit trail.

```
eval/
├── pyproject.toml
├── README.md

│   ├── pii_phi_pci.py                # jurisdiction-tagged eval
│   ├── metrics.py                    # precision/recall/F1/ECE + reliability diagrams
│   ├── latency.py                    # p50/p99/p99.9 + percentile plots
│   ├── cost.py                       # $/1k verdicts calculator (where pricing known)
│   ├── baselines/
│   │   ├── __init__.py
│   │   ├── defenseclaw_regex_only.py
│   │   ├── gpt_oss_120b_judge.py
│   │   └── ai_defense_alone.py
│   ├── report.py                     # generates docs/eval-results.md
│   └── scripts/
│       ├── smoke.py                  # fast subset for CI eval-smoke job
│       └── run_full.py               # full nightly run
└── tests/
```

Output: `eval/results/<git-sha>/` contains:
- `summary.md` — the headline table
- `per_dataset/<dataset>.json` — raw per-prompt verdicts
- `reliability/<evaluator>.png` — calibration diagram
- `latency/<evaluator>.png` — percentile plot
- `cost.json` — cost calculation
- `versus_baselines.md` — Aegis vs the 3 baselines, side-by-side

The nightly `eval.yml` workflow commits `summary.md` → `docs/eval-results.md` on `main` so the README's eval table updates automatically.

## Honesty bar (NO hand-waving in the eval table)

Per the architect lens and Abu's explicit anti-hallucination rule:

- **No "estimated" numbers in the table.** Either we ran the eval and got the number, or we leave it blank.
- **`AEGIS_AI_DEFENSE_MOCK=true` produces a mock-flagged summary.** The mock baseline lets the eval table render with `mock=true` annotations so the demo + CI work even before AI Defense access lands. Real numbers ship in a follow-up commit.
- **Imprompter eval flags Mistral's patch.** README explicitly says "Mistral patched the rendering-layer vector 2024-09-13; we detect the payload pattern in the LLM output for defense-in-depth, not as the sole defense" — per `context/01-threat-landscape/04-pii-and-data-leak-mechanics.md` updated section.
- **MSJ scaling honesty.** The eval table reports detection rate as a function of shot-count; the README explicitly states "we sit on the same power-law scaling curve as Anthropic showed; this is a probabilistic gate, not a deterministic one."
